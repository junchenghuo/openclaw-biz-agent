- 日期：2026-03-07

## 1. 背景与目标
当前需要一个**简单、美观、可本地部署**的登录页，支持用户名+密码登录，调用后端登录接口。

**目标**
- 提供清晰、稳定的登录交互与错误提示
- 与后端接口完成鉴权/会话建立
- 登录成功后按产品设定跳转（需确认）

**非目标**
- 注册/找回密码/验证码/第三方登录（不在本期）
- 权限管理与多租户（不在本期）

## 2. 使用场景与用户故事
- 作为用户，我可以输入用户名与密码并登录系统。
- 作为用户，当我输入不合法或账号密码错误时，我能得到明确提示并可重试。
- 作为用户，我希望页面简洁美观，在桌面端常见分辨率下可正常使用。

## 3. 范围（Scope）
### 3.1 页面
- `/login` 登录页（单页）

### 3.2 表单字段
| 字段 | 类型 | 必填 | 规则（前端校验） | 说明 |
|---|---|---:|---|---|
| username | string | 是 | `trim` 后长度 1~64；不允许全空格 | 用户名/账号 |
| password | string | 是 | 长度 1~128；不自动 trim（保留空格作为密码字符） | 密码 |

> 注：用户名是否允许邮箱/手机号格式，暂不强校验，仅做非空与长度；若后端有更严格规则，以后端返回为准。

### 3.3 按钮/控件
- 主按钮：`登录`
- 辅助：`显示/隐藏密码`（眼睛图标，可选但建议加）
- `记住我`：**待确认**（若后端支持 refresh token/长期会话再做）

## 4. 交互与状态
### 4.1 初始状态
- username、password 为空
- 登录按钮：disabled（当两字段均满足最小校验后 enabled）

### 4.2 输入校验与提示（前端）
触发时机：
- 失焦（blur）时提示字段级错误
- 点击“登录”时若不通过，聚焦到第一个错误字段

字段级错误文案（建议）：
- username 为空：`请输入用户名`
- username 全空格：`用户名不能为空`
- username 过长：`用户名长度不超过 64 个字符`
- password 为空：`请输入密码`
- password 过长：`密码长度不超过 128 个字符`

### 4.3 提交登录
- 点击登录 / 在 password 输入框按 Enter：触发提交
- 提交中：
  - 登录按钮显示 loading（文案可变为“登录中…”）
  - 禁用输入框与按钮（避免重复提交）

### 4.4 成功处理
- 接口返回成功后：
  - 若返回 token：前端按约定写入存储（**存储策略需确认**：HttpOnly Cookie 优先；若是 localStorage/sessionStorage 需评估安全风险）
  - 跳转到：**待确认**（默认建议 `/` 或 `/dashboard`）

### 4.5 失败处理（错误提示分层）
> 原则：尽量给用户可行动的信息；不要暴露安全敏感细节。

- 账号或密码错误（401/业务码）：页面顶部或表单下方展示：`用户名或密码错误`
- 账号被禁用/无权限（403/业务码）：`账号无权限或已被禁用，请联系管理员`
- 参数不合法（400）：
  - 若后端返回字段错误：映射到对应字段下方
  - 否则统一提示：`输入信息有误，请检查后重试`
- 服务异常（5xx/网络超时）：`服务异常，请稍后重试` + 保留重试能力

### 4.6 视觉与布局（“美观”落地）
- 居中卡片式布局（Card）+ 适度留白
- 支持深浅色（可选，至少保证默认主题美观）
- 适配常见桌面分辨率（1366x768~1920x1080）
- Logo/产品名：**待确认**（没有则用占位）

## 5. 接口对接（待确认项先占位）
### 5.1 登录接口
- Method：POST
- URL：`<BASE_URL>/api/login`（待确认）
- Request（示例）：
```json
{
  "username": "alice",
  "password": "******"
}
```
- Response（成功示例）：
```json
{
  "code": 0,
  "data": {
    "token": "...",
    "user": {"id": "1", "name": "Alice"}
  },
  "message": "ok"
}
```
- Response（失败示例）：
```json
{
  "code": 1001,
  "message": "invalid credential"
}
```

### 5.2 会话/鉴权方式（需明确）
- A：后端 Set-Cookie（HttpOnly, Secure, SameSite）建立会话（推荐）
- B：返回 token，前端存储并在后续请求带 `Authorization: Bearer <token>`

## 6. 非功能要求
- 性能：首屏可交互 < 2s（本地部署）
- 可用性：避免重复提交；网络抖动有明确提示
- 安全：
  - 密码输入框不回显；禁止日志打印密码
  - 若使用 localStorage 存 token，需评估 XSS 风险（推荐 Cookie）
- 可观测：前端记录登录成功/失败次数（可选，需确认是否接入埋点）

## 7. 验收标准（AC）
### 7.1 功能
1. 打开 `/login` 能看到居中美观的登录表单，包含用户名、密码、登录按钮（及可选显示/隐藏密码）。
2. 用户名/密码为空时不可提交；点击登录会提示对应字段错误文案。
3. 输入合法后，点击登录或 Enter 触发一次请求；提交中按钮 loading 且禁止重复提交。
4. 后端返回“账号密码错误”时，页面提示 `用户名或密码错误`，且可再次输入并重试。
5. 网络异常/5xx 时提示 `服务异常，请稍后重试`。
6. 登录成功后按约定保存会话/token，并跳转到约定页面。

### 7.2 体验/视觉
7. 在 1366x768 与 1920x1080 下布局不溢出、不遮挡，字体/间距合理。
8. 错误提示位置清晰（字段下方 +/或 顶部提示），不闪烁、不抖动。

### 7.3 安全/质量
9. 前端不在任何日志中输出明文密码。
10. 登录页刷新/后退行为符合预期：
   - 已登录状态访问 `/login`：应自动跳转到目标页（需确认是否实现）。

## 8. 待发起人/Leader 补充确认的问题清单
1. **登录接口信息**：BASE_URL、具体路径、是否需要额外字段（如 captcha、tenant、client_id）。
2. **请求/响应格式**：
   - 成功码/失败码定义？HTTP 状态码是否规范使用？
   - 失败时是否返回字段级错误结构？（用于精准提示）
3. **鉴权方式**：Cookie 会话还是 token？token 是否有 refresh 机制？过期时间？
4. **成功后跳转**：登录成功默认跳转到哪里（`/`、`/dashboard` 或来源页 redirect）？
5. **已登录访问登录页**：是否要求自动跳转（防止重复登录）？
6. **UI 资产**：是否有 logo/产品名/主题色？没有的话是否接受通用模板。
7. **本地部署形态**：
   - 前端与后端是否同域？是否有跨域（CORS）问题？
   - 是否需要配置 `.env` 的 API 地址？
8. **账号体系**：用户名是工号/邮箱/手机号/自定义？长度/字符集是否有限制？
