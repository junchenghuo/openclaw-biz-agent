# 科技风登录页（含图形验证码）PRD（Gate-1）

- 项目：`projects/tech-login-captcha-page`
- 形态：PC 为主，兼顾移动端适配
- 技术约束：沿用现有体系，不引入新技术栈
- 版本：v1.0（Gate-1 评审稿）
- 负责人：产品（楚轩）

---

## 1. 背景与问题
当前系统需要一个“更可信、更符合等保/安全要求”的登录入口：
- 账号+密码的传统登录存在被撞库/脚本暴力尝试风险
- 需要加入**后端生成的图形验证码**以增加自动化攻击成本
- 同时希望登录页具备“科技感”视觉（深色科技蓝、玻璃质感、粒子背景、霓虹按钮），以统一产品形象

## 2. 目标（Goals）
1. 提供稳定可用的登录页：账号 + 密码 + 图形验证码。
2. 通过验证码与策略降低脚本攻击与重复尝试。
3. 兼容 PC/移动端（响应式），核心交互在移动端可用。
4. 支持“记住我”（Remember me）以改善体验（在安全策略允许前提下）。

## 3. 非目标（Non-goals）
- 不做注册/找回密码/第三方登录（本期不在范围）。
- 不引入短信/邮件/OTP/2FA（可作为后续增强）。
- 不在本期强制实现复杂的风控（设备指纹等），但预留扩展点。

## 4. 用户与使用场景
- 内部员工/客户管理员：日常登录进入系统。
- 典型场景：
  - 首次登录/久未登录，需要输入验证码
  - 连续输错密码或验证码，需要提示并可刷新验证码
  - 弱网/接口异常时需要明确错误与重试

## 5. 成功指标（Metrics）
- 登录成功率（排除密码错误后）：≥ 99.5%
- 页面可用性：核心资源加载成功率 ≥ 99.9%
- 关键接口 p95：
  - `/captcha` p95 ≤ 300ms（不含图片下载 CDN 的情况下按现有链路）
  - `/login` p95 ≤ 500ms（不含后续跳转资源加载）
- 安全侧（观察指标）：单位时间登录尝试异常峰值下降、验证码命中率/刷新率可追踪

## 6. 信息架构与页面结构
### 6.1 页面布局（PC）
- 全屏背景：深色科技蓝渐变 + 动态粒子（可降级为静态图）
- 中央登录卡片（玻璃拟态）：
  - Logo/产品名（支持替换）
  - 文案：主标题（中文）+ 副标题（可选）
  - 表单：账号、密码、验证码（图+输入框+刷新）
  - 记住我（checkbox）
  - 登录按钮（主按钮、霓虹光效）
  - 底部辅助：语言切换入口（占位）、版权信息

### 6.2 页面布局（移动端）
- 登录卡片居中/靠上，宽度自适应（建议 16px 边距）
- 验证码区域保持可点可刷新，避免过小导致误触

## 7. 功能需求（Functional Requirements）

### 7.1 表单字段定义与校验
| 字段 | 类型 | 必填 | 规则 | 错误提示（示例） |
|---|---|---:|---|---|
| 账号（username） | input text | 是 | 去首尾空格；长度 1-64；允许邮箱/手机号/用户名（按现有规则） | 请输入账号 |
| 密码（password） | input password | 是 | 长度 1-64（或沿用现有）；不做前端复杂校验 | 请输入密码 |
| 验证码（captchaCode） | input text | 是 | 去空格；建议统一转大写（如后端不区分大小写）；长度 4-6（由后端返回） | 请输入验证码 |
| 记住我（rememberMe） | checkbox | 否 | 默认关闭 | - |

补充交互：
- Enter 键触发登录（在输入框聚焦时）。
- 登录中（loading）禁止重复提交。

### 7.2 验证码展示与刷新策略
**展示**：验证码以图片形式展示（base64 或图片 URL），同时携带 `captchaId`（或 token）。

**刷新触发**：
1. 用户点击验证码图片或“换一张”按钮；
2. 登录失败且错误码为「验证码错误/验证码过期/需要验证码」时，前端自动刷新；
3. 页面初次进入时自动拉取一次。

**失效策略（约定）**：
- 验证码有效期：建议 60s～180s（以安全要求为准）；
- 验证码校验失败次数：后端可按策略限制（例如同一个 captchaId 允许 3 次尝试）。

**前端表现**：
- 刷新中：验证码区域显示 skeleton/小 loading；
- 刷新失败：提示“验证码加载失败，点击重试”。

### 7.3 登录流程
1. 进入页面 → 调用 `/captcha` 获取 `captchaId + captchaImage`。
2. 用户输入账号/密码/验证码 → 点击登录。
3. 前端调用 `/login`，携带：username、password（按现有加密/传输规范）、captchaId、captchaCode、rememberMe。
4. 登录成功：
   - 存储 token/session（按现有体系）
   - 跳转到系统首页或上次访问页面（如有 redirect 参数）
5. 登录失败：根据错误码展示提示并决定是否刷新验证码。

### 7.4 Loading 状态
- 点击登录后：按钮显示 loading，表单禁用。
- 若超过 8s 未返回：提示“网络较慢，请稍后重试”，保留 loading 但允许用户取消/重试（实现可选；至少要可再次点击登录）。

### 7.5 错误态与提示文案
**通用原则**：不暴露过多敏感信息（如“账号不存在”可与“密码错误”统一为“账号或密码错误”，按现有安全规范）。

建议错误码到 UI 行为映射：
| 场景 | 后端错误码（示例） | 提示 | 是否刷新验证码 |
|---|---|---|---:|
| 账号或密码错误 | AUTH_INVALID_CREDENTIALS | 账号或密码错误 | 是（可选，建议是） |
| 验证码错误 | CAPTCHA_INVALID | 验证码错误，请重试 | 是 |
| 验证码过期 | CAPTCHA_EXPIRED | 验证码已过期，请重新获取 | 是 |
| 频率限制 | RATE_LIMITED | 操作过于频繁，请稍后再试 | 是（建议否，避免刷） |
| 服务异常 | SERVER_ERROR | 服务开小差了，请稍后重试 | 否 |
| 网络错误/超时 | NETWORK_ERROR | 网络异常，请检查网络后重试 | 否 |

错误展示位置：
- 优先表单顶部一个全局错误条（toast/alert），同时对具体字段做红框与提示（如验证码为空）。

### 7.6 “记住我”规则
**产品侧建议**（以安全/等保要求为准）：
- rememberMe=true：延长登录态有效期（例如 7 天/30 天），或使用 refresh token；
- rememberMe=false：短会话（例如 2 小时/浏览器关闭失效，按现有策略）。

前端存储：沿用现有 token 存储策略（如已规定 localStorage/cookie/httpOnly cookie）。
> 若现有体系支持 httpOnly cookie，优先该方案；否则明确安全风险与缓解（见「安全与合规」）。

### 7.7 i18n/语言预留
- 文案中文默认；预留语言切换入口（UI 占位即可）。
- 所有文案走统一的 i18n key（若现有项目已接入）。

## 9. 接口契约草案（后端）
> 目标：把前后端对齐到“可联调”的最小契约；字段名可按现有后端规范调整。

### 9.1 获取验证码 `/captcha`
- Method：GET
- Auth：无（未登录）
- Query：
  - `w`（可选）：期望宽度
  - `h`（可选）：期望高度
  - `t`（可选）：时间戳/随机数，避免缓存

**Response 200**
```json
{
  "captchaId": "cpt_1234567890",
  "imageType": "png",
  "image": "data:image/png;base64,iVBORw0KGgoAAA...",
  "expireIn": 120,
  "codeLength": 4,
  "caseSensitive": false
}
```

**Response 错误**
- 429：限流
- 5xx：服务异常

### 9.2 登录 `/login`
- Method：POST
- Auth：无（未登录）
- Body：
```json
{
  "username": "zhangsan",
  "password": "<按现有传输规范>",
  "captchaId": "cpt_1234567890",
  "captchaCode": "A7K2",
  "rememberMe": true,
  "redirect": "/" 
}
```

**Response 200**（示例，按现有体系返回）
```json
{
  "token": "jwt_or_session_token",
  "tokenType": "Bearer",
  "expireIn": 7200,
  "refreshToken": "optional",
  "user": {
    "id": "u_1",
    "name": "张三"
  }
}
```

**Response 4xx（示例）**
```json
{
  "code": "CAPTCHA_INVALID",
  "message": "captcha invalid"
}
```

错误码建议集合：
- `AUTH_INVALID_CREDENTIALS`
- `CAPTCHA_INVALID`
- `CAPTCHA_EXPIRED`
- `RATE_LIMITED`
- `SERVER_ERROR`

## 10. 非功能需求（NFR）
- 可用性：页面在接口异常时可重试，不卡死。
- 性能：验证码图建议走缓存友好的方式（但需防止缓存导致复用）。
- 可访问性：表单 label 完整，键盘可操作，错误提示可读。
- 兼容：主流 Chrome/Edge；移动端 Chrome/Safari（如产品要求）。

---

## 11. 安全与合规要点
- 不在错误提示中区分“账号不存在/密码错误”（若现有安全规范要求）。
- 验证码必须由后端生成并校验，不允许前端生成。
- 传输层必须 HTTPS。
- 若 token 存储在 localStorage：
  - 需评估 XSS 风险；建议结合 CSP/输入输出过滤/安全扫描；
  - 若现有体系支持 cookie(httpOnly+sameSite) 优先。
- 增强建议（可选）：连续失败次数触发更严格策略（更高难度验证码/延时/锁定）。

---

## 12. 依赖与风险
### 依赖
- 后端提供 `/captcha` 与 `/login` 对应接口与错误码。
- 前端沿用现有登录态管理与路由跳转逻辑。
- 设计资源：科技风视觉稿（若 UI 需要落地高保真）。

### 风险
- 验证码刷新过频可能造成接口压力（需限流与前端节流）。
- 记住我与等保策略可能冲突（需要明确会话策略与审计）。

---

## 13. 验收清单（可测项）
1. 页面首次进入自动拉取验证码；失败可点击重试。
2. 账号/密码/验证码为空时：字段提示正确，禁止提交或提交后提示。
3. 输入正确账号+密码+正确验证码：登录成功并跳转。
4. 验证码错误：提示正确，验证码自动刷新，且原验证码不可复用。
5. 验证码过期：提示正确，验证码自动刷新。
6. 连续点击登录：loading 状态下不可重复提交。
7. 网络断开/超时：提示网络异常，可重试。
8. 记住我开启/关闭：对应登录态有效期符合后端策略（以实际实现为准）。
9. 移动端：布局不溢出；验证码与按钮可点击；键盘弹出不遮挡关键按钮（尽量）。
10. i18n 占位：文案可被替换为英文（至少 key 可配置）。
