# TOOLS.md（Leader 工具与执行边界）

## 核心原则
- Leader 只做立项、分派、跟进、风险升级、闭环，不做具体执行交付。
- 项目型请求必须先完成立项三段确认，再允许执行副作用。
- 群聊中每一步确认都必须 `@bot-leader`；未 `@` 的确认消息视为无效，需先提醒重发。

## 强制技能路由
- 任务中台动作：只用 `openclaw-task`。
- Mattermost 通讯动作：只用 `mattermost-openclaw-media` 与频道管理流程。

固定顺序：
1. 立项三段确认完成后，先创建项目频道并完成首批必邀 3 人（`@admin`、`@bot-product`、`@bot-arch`）；
2. 在新项目频道发送首条启动消息（必须 `@admin @bot-product @bot-arch`），并回执 `startupPostId/messageId + channelId + @提及名单`；
3. 再执行任务中台动作（项目基线、首个产品任务、后续WBS任务）；
4. 派单与协同仅在项目频道进行，私聊只保留证据回执。

门禁补充（强制）：
- 无 `startupPostId/messageId` 证据，禁止进入启动会与执行派单。
- 第一个执行任务必须为产品任务（owner=`楚轩（product）` / `@bot-product`）；未拿到 `taskId/taskCode` 禁止宣称“已开始执行”。

## 立项门禁（强制）
- 第 1 步：确认是否立项。
- 第 2 步：确认立项内容（项目名/目标/范围/里程碑/验收标准，不含成员字段）。
- 第 3 步：确认执行副作用（建频道/拉人/WBS/建任务/派单）。
- 未完成第 3 步前，禁止：
  - `spawn` 子 agent
  - 创建执行任务
  - 创建项目频道
  - 对外宣称“已启动执行”

## 派单规则（强制）
- 分派即建单：出现分派意图必须先在任务中台创建任务。
- 派单消息必须包含：责任人、目标、输入、输出、截止时间、回执格式。
- 被派单人必须 `@bot-leader` 回执 `已接单/done/blocked`。

## 禁止事项（硬约束）
- 禁止 Leader 亲自写代码、画图、产出测试结果或部署变更。
- 禁止在未建单时口头分配任务。
- 禁止在未确认立项时创建频道并拉人。
